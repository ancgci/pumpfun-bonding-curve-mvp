## Diagnóstico

### Causa Raiz: `recheckMaxAttempts` não configurado no modo AGGRESSIVE

O arquivo `data/ta-config.json` define o modo de operação como `AGGRESSIVE` (scoreMinimo=1 — ultra-permissivo). Porém, os parâmetros de recheck **não estavam declarados** neste modo:

```json
"AGGRESSIVE": {
  "scoreMinimo": 1,
  // recheckMaxAttempts → NÃO DEFINIDO → padrão do código: 1
  // recheckDelayMs     → NÃO DEFINIDO → padrão do código: 6000ms
}
```

O código (`agentOrchestrator.ts`) usa estes valores para o loop de recheck do `checkEntryBlocks`:

```typescript
const recheckDelayMs = Math.max(1000, taConfigExec.recheckDelayMs || 6000);
const recheckMaxAttempts = Math.max(1, taConfigExec.recheckMaxAttempts || 1);
```

Com **apenas 1 tentativa de 6 segundos**, o bot não conseguia acumular dados de TA em tempo real (candles de 1s) suficientes para o `getTASnapshotV2` retornar um snapshot válido. O bloco `BLOCK_INSUFFICIENT_DATA` persistia dentro da janela de recheck, causando timeout sistemático.

### Por que o modo SIMULATION funcionava normalmente?

O modo **SIMULATION** usa o caminho `runProtocolSimulationDiscovery`, que **não executa o `checkEntryBlocks`** — vai diretamente para a decisão LLM e registro simulado. O modo **LIVE** usa `processPumpFunObservation`, que executa o **pipeline completo de 8 estágios**, incluindo o recheck de hard blocks no Step 5.

## Bug #2 — `tOutput.type` inválido no executor (`index.ts:1608`)

### Descrição

Ao corrigir o Bug #1, o pipeline passou a fluir até o Pipeline 8. No executor, porém, outro problema se revelou:

```
💰 [LIVE] Executing real trade...
🔄 Executando trade híbrido para token ... (Tipo: SELL, Force: false)
⚠️ [LIVE] Trade não executado para Phantom: SELL_POSITION_NOT_FOUND
```

O executor recebia `Tipo: SELL` em vez de `Tipo: BUY`, causando falha na execução.

### Causa Raiz

Na função `processPumpFunObservation` (`index.ts:1608`), o closure `executeTradeWithRetry` usava **o tipo da transação de mercado observada** (`tOutput.type`) em vez do **tipo da decisão do agente** (sempre `"BUY"` neste ponto do pipeline):

```typescript
// ANTES (bugado):
const executionResult = await executeHybridTrade(tokenData, tOutput.type, force, buyAmountSol);
//                                                            ^^^^^^^^^^
//                              Pode ser "SELL" se o evento que disparou a descoberta foi uma venda

// DEPOIS (corrigido):
const executionResult = await executeHybridTrade(tokenData, "BUY", force, buyAmountSol);
// FIX 2026-04-16: use agente decision type, not raw market event type
```

**Impacto:** Quando um evento SELL de mercado (ex: uma baleia vendendo) acionava a descoberta de um token a 90%+, o `tOutput.type = "SELL"` chegava ao executor. O executor tentava VENDER uma posição inexistente → `SELL_POSITION_NOT_FOUND`.

### Correção

- **Arquivo:** `/home/anto/pumpfun-bot/index.ts` (linha 1608)  
- **Backup criado:** `index.ts.bak.<timestamp>`  
- Bot reiniciado via `pm2 restart bot` às **10:39 do dia 16/04/2026**

## Arquitetura do Sistema (Entendimento Pós-Diagnóstico)

### Threshold 97.7% — Design Intencional

O `executeHybridTrade` só executa compras quando `curvePercent >= 97.7%` (`hybridExecutor.ts:1223`):

```typescript
const isDiscoveryBuy = tokenData.mode === "CURVE" && tokenData.curvePercent >= 97.7;
```

O fluxo correto é:
1. **Descoberta** a 90-97.6% → Pipeline completo roda → `BUY_NOT_ELIGIBLE:CURVE:X` → DipWaitlist
2. **Token avança** até 97.7%+ → novo evento dispara `processPumpFunObservation` 
3. **Re-pipeline** (mint não está em `aiProcessedAddresses`, pois `persistDecision=false`) → executor aprova → **COMPRA EXECUTADA**

Tokens com `BUY_NOT_ELIGIBLE:CURVE:X` nos logs são **comportamento esperado**, não um erro.
